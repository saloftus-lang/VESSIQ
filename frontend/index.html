The Problem

Maritime data is fragmented.
Your ops team shouldn't fix it.

Every carrier, terminal, and port uses a different format and field naming convention. Reconciling it manually costs hours daily — and creates SLA risk.

01

Format fragmentation

EDI X12, EDIFACT, CSV, XML, PDF — every source has its own structure. No two carriers share the same field names or event codes.

02

Manual reconciliation

Ops teams spend hours per day stitching together vessel events from carrier portals, TOS exports, and email advisories before they can act.

03

Downstream system failures

TMS and visibility platforms require clean, consistent data. Bad input cascades into incorrect ETAs, missed updates, and demurrage exposure.

Capabilities

Built for production maritime workflows

Everything a logistics operator needs to stop wrestling with raw data.

EDI 315 Translation

Full X12 315 parser — multi-transaction envelopes, ISA/GS/ST segments, B4 event codes, R4 port locations, and DTM timestamps across all major carrier formats.

Terminal CSV Normalization

Ingests TOS exports from Navis N4, SPARCS, and custom systems. YAML-driven column mapping means new terminals take minutes, not sprints.

Unified VesselEvent Schema

Every event maps to the same object: vessel, voyage, container, event type, port, timestamps, and references — regardless of source format.

Filterable Event API

Query by vessel, voyage, port LOCODE, container, event type, or source. Paginated, fast, and ready to power dashboards or downstream feeds.

Strict Schema Validation

Every event validated by Pydantic v2. Malformed records are rejected with clean error detail — never silently dropped or corrupted downstream.

Config-Driven Extensibility

Add a new carrier or terminal by editing a YAML mapping file. No Python required. New data sources ship in hours, not engineering sprints.
